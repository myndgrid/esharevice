# Feature: Edit exchange item

**Created:** 2026-05-16 05:30 UTC
**Last Updated:** 2026-05-16 05:30 UTC
**Status:** Live at `https://esharevice.com/items/[id]/edit` for the owner of any item.

Closes the listing-lifecycle loop. A user who posted an exchange item can now edit it — change the structured fields, swap the photo, or both. Non-owners are bounced server-side (both at the page render and at the API).

## Routes / API Endpoints

| Method | Path | Auth | Description |
|---|---|---|---|
| PUT | `/v1/exchange-items/{id}` | Bearer + owner + Idempotency | Partial update of the row. All fields optional; only keys actually present in the body are written so a client can patch one field without clearing others |
| (page) | `/items/[id]/edit` | session + owner | Server component; redirects non-owners to the detail page (defense in depth — API also 403s) |
| (server action) | `editItemAction(itemId, …)` | session + owner | Validates with `ExchangeItemUpdate`, PUTs to the API, then optionally re-runs the image upload pipeline with a derived idempotency key |

## Persistence (tables touched)

- **Updated:** `exchange_items` — partial update of provider / service / date / exchange / description / rate_type / updated_at. `img_key + img_hash` updated via the existing image-upload pipeline if a new photo is selected.
- No new tables or columns.

## Edge Cases & Gotchas

- **Owner-only at two layers.** Page-level redirect (so non-owners don't see a useless pre-filled form) + API-level 403 (so direct PUT calls are rejected). Either alone would be enough; both gives defense in depth.
- **Empty-string vs undefined.** FormData fields default to empty strings when a user clears them. The server action's `takeOrUndefined` helper collapses empty strings to `undefined` so the API doesn't accidentally clear optional fields (e.g. `rate_type`). Mandatory fields (provider/service/etc.) are `required` on the input element so the browser blocks submit before this matters.
- **Date input parses ISO only.** The API stores `date` as a free-form string ("next Friday" is legal). The form's `<input type="date">` only populates when the stored value parses as `new Date()`. Free-form dates stay editable via the form's pristine fallback to empty.
- **Image replacement is optional.** The picker is empty by default and shows the existing photo as a preview. Submitting without picking keeps the current image; submitting with a new image triggers the upload pipeline (sha256 dedup means swapping back to an old image costs zero new storage).
- **Idempotency key shared across the two API calls.** PUT uses `<key>`; image upload uses `<key>-image`. Different paths → different fingerprints → no collision. Replay of either call is safe and produces no duplicate row update or duplicate variant upload.
- **Cache invalidation.** `revalidatePath("/items/[id]")` + `revalidatePath("/")` so the home page and the detail page show the new values on the next render. Without this, both would show stale data until their cache TTLs expired.
- **Image upload failure post-update.** Same recovery pattern as create — if the API update succeeds but the image upload fails, the row is kept and the user is redirected to the detail page with `?image_error=<reason>`. They can re-submit the edit to retry.
