# Feature: Messages (conversations + messages, phase A)

**Created:** 2026-05-16 07:30 UTC
**Last Updated:** 2026-05-16 09:15 UTC
**Status:** Live with real-time SSE. Phase A (REST) + Phase B-1 (SSE through Caddy with 25 s heartbeats) shipped. Phase B-2 (email-on-new-message) still deferred â€” current behaviour is "must have the thread open to see new messages live."

A non-owner can open any item's detail page, tap **Message owner**, and start a thread with the listing's owner. Both parties see the thread on `/messages` and exchange messages in `/messages/[id]`. New messages from the other side appear within ~5 s thanks to the client poll loop.

## Routes / API Endpoints

| Method | Path | Auth | Description |
|---|---|---|---|
| POST | `/v1/exchange-items/{id}/conversations` | Bearer + Idempotency | Create-or-get the conversation between viewer + owner. 201 on first call, 200 on subsequent. 403 if viewer IS the owner. UNIQUE (item_id, initiator_id) makes the operation idempotent at the SQL layer. |
| GET | `/v1/conversations` | Bearer | All threads viewer participates in (as initiator OR as the item owner). Cursor-paginated by `last_message_at DESC`. Joined with items + users + most-recent message for the list-view payload. |
| GET | `/v1/conversations/{id}` | Bearer | Single thread metadata. 403 if not a participant. |
| GET | `/v1/conversations/{id}/messages` | Bearer | Oldest-first message page. Forward-cursor on `(created_at, id)` so the poll loop fetches only what's new. |
| POST | `/v1/conversations/{id}/messages` | Bearer + Idempotency | Append a message; bumps `conversations.last_message_at`; publishes to the in-process bus so any open SSE subscriber on the conversation receives it within an event-loop tick. |
| GET | `/v1/conversations/{id}/events` | Bearer | SSE stream. `event: ready` on open, `event: message` per new message (data = full `Message` JSON), `event: heartbeat` every 25 s. Cleanup (EventEmitter unsubscribe + heartbeat clear) on client abort. |

## Persistence (tables touched)

- **New:** `conversations` (`id`, `item_id` FK, `initiator_id` FK, `created_at`, `last_message_at`). UNIQUE (item_id, initiator_id). Indexes on `(initiator_id, last_message_at DESC)` and `(item_id)`.
- **New:** `messages` (`id`, `conversation_id` FK, `sender_id` FK, `body` (1-4000 chars, CHECK), `created_at`). Index on `(conversation_id, created_at DESC, id DESC)` supports both chat pagination and the latest-per-conversation DISTINCT ON.
- Both FK chains cascade â€” deleting a user / item / conversation cleans up its messages without orphans.

## Edge Cases & Gotchas

- **Idempotent thread creation.** `UNIQUE (item_id, initiator_id)` means re-clicking "Message owner" lands on the same thread. The handler does `INSERT â€¦ ON CONFLICT DO NOTHING + RETURNING` then falls back to a `SELECT` when nothing was inserted. Returns 201 vs 200 accordingly so a Sentry breadcrumb / log can tell the cases apart.
- **Polling cursor.** The client encodes the last-message tuple `(created_at, id)` as base64 and sends it on every poll. The API's forward-cursor returns only newer rows; an idle thread costs a single empty page every 5 s (negligible).
- **Optimistic-UI dedup.** Local `tempId` is the idempotency key. If the API succeeds, we swap the optimistic row for the canonical one (same `tempId` slot, different DB `id`). The poll loop's `mergeNew` then dedups by id so a server-side echo from the polling fetch can't double-insert.
- **Server actions for the client fetch path.** Required because `lib/api` transitively imports `next/headers`. The wrapping is purely a Next.js bundling concern â€” same code, just routed via server-action POST under the hood.
- **`force-dynamic` on every page** because both `requireAuth` and `api.me()` read cookies â€” any cache would serve stale auth state.
- **Owner-side messaging.** The owner's `/messages` listing surfaces threads started by non-owners (`exchange_items.user_id = me`). The owner can reply but can NOT initiate a new thread on their own listing â€” `POST /v1/exchange-items/{id}/conversations` returns 403 if `viewer == owner`. By design â€” there's no "broadcast to all reservers" semantic yet.
- **Cascading deletes.** Hard-deleting an exchange_item (not the normal flow â€” the app uses soft-delete via `archived_at`) would cascade to its conversations + messages. Archive leaves both intact; the API just hides the item from listings, so existing threads stay accessible for both parties' history.

## Real-time architecture (Phase B-1 â€” live)

- **In-process pub/sub.** [apps/api/src/lib/message-bus.ts](../../apps/api/src/lib/message-bus.ts) is a thin EventEmitter keyed by `conversation_id`. Publish on insert, subscribe in the SSE route. Single-container design; Redis pub/sub is the obvious next step if we ever go multi-replica.
- **SSE route.** `GET /v1/conversations/{id}/events` (auth + participant check). Initial `event: ready` frame flushes headers + signals open. Then one `event: message` per publish with the full `Message` row as data. 25 s heartbeat keeps Caddy's default 60 s idle-proxy-timeout from killing quiet threads (bug-registry: `[Network] Long-Running Connection Dropped by Proxy / Load Balancer`).
- **Caddy pass-through.** [infra/Caddyfile](../../infra/Caddyfile) splits `api.{$DOMAIN}` into two `handle` blocks: `/v1/.../events` goes through `flush_interval -1` with no compression (gzip would buffer); everything else keeps `encode zstd gzip` for JSON throughput.
- **Same-origin proxy on the web.** [apps/web/app/api/messages/[id]/events/route.ts](../../apps/web/app/api/messages/[id]/events/route.ts) bridges browser `EventSource` (no header support) to the API's Bearer-authed endpoint. The proxy reads the session cookie server-side, opens the upstream with the access token, pipes the response body, and forwards client abort to the upstream fetch so the API-side cleanup fires.
- **Client.** [apps/web/app/messages/[id]/conversation-view.tsx](../../apps/web/app/messages/[id]/conversation-view.tsx) opens `EventSource('/api/messages/:id/events')` on mount. `ready` â†’ catch-up fetch with the tail cursor (covers the page-render-to-SSE-open race); `message` â†’ merge into state with the existing dedup. EventSource handles reconnect natively; on each reconnect the next `ready` event triggers another catch-up.
- **Fallback poll.** If `EventSource` never opens (network blocked, CDN buffering, proxy stripping), the same `fetchMessagesAfterAction` cursor fetch runs every 30 s. Both feeds share `mergeNew` so no dedupe handling needed at the consumer.
