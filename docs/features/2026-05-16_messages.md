# Feature: Messages (conversations + messages, phase A)

**Created:** 2026-05-16 07:30 UTC
**Last Updated:** 2026-05-16 07:30 UTC
**Status:** Live. Phase A = REST endpoints + a polling-based web client (5 s tick). Phase B (SSE through Caddy with 25 s heartbeats + email-on-new-message) deferred to a follow-up.

A non-owner can open any item's detail page, tap **Message owner**, and start a thread with the listing's owner. Both parties see the thread on `/messages` and exchange messages in `/messages/[id]`. New messages from the other side appear within ~5 s thanks to the client poll loop.

## Routes / API Endpoints

| Method | Path | Auth | Description |
|---|---|---|---|
| POST | `/v1/exchange-items/{id}/conversations` | Bearer + Idempotency | Create-or-get the conversation between viewer + owner. 201 on first call, 200 on subsequent. 403 if viewer IS the owner. UNIQUE (item_id, initiator_id) makes the operation idempotent at the SQL layer. |
| GET | `/v1/conversations` | Bearer | All threads viewer participates in (as initiator OR as the item owner). Cursor-paginated by `last_message_at DESC`. Joined with items + users + most-recent message for the list-view payload. |
| GET | `/v1/conversations/{id}` | Bearer | Single thread metadata. 403 if not a participant. |
| GET | `/v1/conversations/{id}/messages` | Bearer | Oldest-first message page. Forward-cursor on `(created_at, id)` so the poll loop fetches only what's new. |
| POST | `/v1/conversations/{id}/messages` | Bearer + Idempotency | Append a message; bumps `conversations.last_message_at`. |

## Persistence (tables touched)

- **New:** `conversations` (`id`, `item_id` FK, `initiator_id` FK, `created_at`, `last_message_at`). UNIQUE (item_id, initiator_id). Indexes on `(initiator_id, last_message_at DESC)` and `(item_id)`.
- **New:** `messages` (`id`, `conversation_id` FK, `sender_id` FK, `body` (1-4000 chars, CHECK), `created_at`). Index on `(conversation_id, created_at DESC, id DESC)` supports both chat pagination and the latest-per-conversation DISTINCT ON.
- Both FK chains cascade — deleting a user / item / conversation cleans up its messages without orphans.

## Edge Cases & Gotchas

- **Idempotent thread creation.** `UNIQUE (item_id, initiator_id)` means re-clicking "Message owner" lands on the same thread. The handler does `INSERT … ON CONFLICT DO NOTHING + RETURNING` then falls back to a `SELECT` when nothing was inserted. Returns 201 vs 200 accordingly so a Sentry breadcrumb / log can tell the cases apart.
- **Polling cursor.** The client encodes the last-message tuple `(created_at, id)` as base64 and sends it on every poll. The API's forward-cursor returns only newer rows; an idle thread costs a single empty page every 5 s (negligible).
- **Optimistic-UI dedup.** Local `tempId` is the idempotency key. If the API succeeds, we swap the optimistic row for the canonical one (same `tempId` slot, different DB `id`). The poll loop's `mergeNew` then dedups by id so a server-side echo from the polling fetch can't double-insert.
- **Server actions for the client fetch path.** Required because `lib/api` transitively imports `next/headers`. The wrapping is purely a Next.js bundling concern — same code, just routed via server-action POST under the hood.
- **`force-dynamic` on every page** because both `requireAuth` and `api.me()` read cookies — any cache would serve stale auth state.
- **Owner-side messaging.** The owner's `/messages` listing surfaces threads started by non-owners (`exchange_items.user_id = me`). The owner can reply but can NOT initiate a new thread on their own listing — `POST /v1/exchange-items/{id}/conversations` returns 403 if `viewer == owner`. By design — there's no "broadcast to all reservers" semantic yet.
- **Cascading deletes.** Hard-deleting an exchange_item (not the normal flow — the app uses soft-delete via `archived_at`) would cascade to its conversations + messages. Archive leaves both intact; the API just hides the item from listings, so existing threads stay accessible for both parties' history.

## Real-time roadmap (Phase B, deferred)

- Replace the 5 s poll with SSE through Caddy. The bug-registry covers the 25 s heartbeat needed to keep the connection alive past Caddy's idle timeout.
- Email notification when the other party messages and the recipient isn't actively viewing the thread. Same Resend pipeline as week-5 reserve-email; need a "last-seen-conversation-at" timestamp on the participant row OR an "unread count" derived from `messages.created_at > viewer.last_read_at`.
