# Feature: Messages (conversations + messages, phase A)

**Created:** 2026-05-16 07:30 UTC
**Last Updated:** 2026-05-16 14:55 UTC
**Status:** Live end-to-end. Phase A (REST), Phase B-1 (SSE), Phase B-2 (email-on-new-message with active-view suppression), and Phase B-3 (unread-message badge on the Messages tab) all shipped. New behaviour: a recipient who has the thread open in the last 2 minutes is NOT emailed; everyone else gets a Resend transactional email with subject `${senderName}: ${preview}` and a deep-link back to `/messages/[id]`. The mobile tab bar shows a red 9+ badge on the Messages icon reflecting total unread messages across all threads.

A non-owner can open any item's detail page, tap **Message owner**, and start a thread with the listing's owner. Both parties see the thread on `/messages` and exchange messages in `/messages/[id]`. New messages from the other side appear within ~5 s thanks to the client poll loop.

## Routes / API Endpoints

| Method | Path | Auth | Description |
|---|---|---|---|
| POST | `/v1/exchange-items/{id}/conversations` | Bearer + Idempotency | Create-or-get the conversation between viewer + owner. 201 on first call, 200 on subsequent. 403 if viewer IS the owner. UNIQUE (item_id, initiator_id) makes the operation idempotent at the SQL layer. |
| GET | `/v1/conversations` | Bearer | All threads viewer participates in (as initiator OR as the item owner). Cursor-paginated by `last_message_at DESC`. Joined with items + users + most-recent message for the list-view payload. |
| GET | `/v1/conversations/unread-count` | Bearer | `{ total }` — count of messages newer than the viewer's per-conversation `*_last_read_at` AND not authored by themselves. Single integer; drives the Messages tab badge. Registered before `{id}` in the route file so Hono's radix tree resolves the static path first. |
| GET | `/v1/conversations/{id}` | Bearer | Single thread metadata. 403 if not a participant. |
| GET | `/v1/conversations/{id}/messages` | Bearer | Oldest-first message page. Forward-cursor on `(created_at, id)` so the poll loop fetches only what's new. |
| POST | `/v1/conversations/{id}/messages` | Bearer + Idempotency | Append a message; bumps `conversations.last_message_at` AND the sender's `*_last_read_at`; publishes to the in-process bus; fires a fire-and-forget transactional email to the recipient unless their `*_last_read_at` is within 2 minutes of `now()`. |
| PATCH | `/v1/conversations/{id}/read` | Bearer | Mark the viewer's side of the thread as read (`initiator_last_read_at` or `owner_last_read_at = now()`). 204 No Content. Idempotent — safe to fire on every SSE event, every page open, every focus event. Client fires it on SSE-open and on every incoming live message. |
| GET | `/v1/conversations/{id}/events` | Bearer | SSE stream. `event: ready` on open, `event: message` per new message (data = full `Message` JSON), `event: heartbeat` every 25 s. Cleanup (EventEmitter unsubscribe + heartbeat clear) on client abort. |

## Persistence (tables touched)

- **New:** `conversations` (`id`, `item_id` FK, `initiator_id` FK, `created_at`, `last_message_at`, `initiator_last_read_at`, `owner_last_read_at`). UNIQUE (item_id, initiator_id). Indexes on `(initiator_id, last_message_at DESC)` and `(item_id)`. The two `*_last_read_at` columns power per-participant active-view suppression for the email-on-new-message path; both default `NULL` (= "never seen, always email").
- **New:** `messages` (`id`, `conversation_id` FK, `sender_id` FK, `body` (1-4000 chars, CHECK), `created_at`). Index on `(conversation_id, created_at DESC, id DESC)` supports both chat pagination and the latest-per-conversation DISTINCT ON.
- Both FK chains cascade — deleting a user / item / conversation cleans up its messages without orphans.

## Edge Cases & Gotchas

- **Idempotent thread creation.** `UNIQUE (item_id, initiator_id)` means re-clicking "Message owner" lands on the same thread. The handler does `INSERT … ON CONFLICT DO NOTHING + RETURNING` then falls back to a `SELECT` when nothing was inserted. Returns 201 vs 200 accordingly so a Sentry breadcrumb / log can tell the cases apart.
- **Polling cursor.** The client encodes the last-message tuple `(created_at, id)` as base64 and sends it on every poll. The API's forward-cursor returns only newer rows; an idle thread costs a single empty page every 5 s (negligible).
- **Optimistic-UI dedup.** Local `tempId` is the idempotency key. If the API succeeds, we swap the optimistic row for the canonical one (same `tempId` slot, different DB `id`). The poll loop's `mergeNew` then dedups by id so a server-side echo from the polling fetch can't double-insert.
- **Server actions for the client fetch path.** Required because `lib/api` transitively imports `next/headers`. The wrapping is purely a Next.js bundling concern — same code, just routed via server-action POST under the hood.
- **`force-dynamic` on every page** because both `requireAuth` and `api.me()` read cookies — any cache would serve stale auth state.
- **Owner-side messaging.** The owner's `/messages` listing surfaces threads started by non-owners (`exchange_items.user_id = me`). The owner can reply but can NOT initiate a new thread on their own listing — `POST /v1/exchange-items/{id}/conversations` returns 403 if `viewer == owner`. By design — there's no "broadcast to all reservers" semantic yet.
- **Cascading deletes.** Hard-deleting an exchange_item (not the normal flow — the app uses soft-delete via `archived_at`) would cascade to its conversations + messages. Archive leaves both intact; the API just hides the item from listings, so existing threads stay accessible for both parties' history.

## Real-time architecture (Phase B-1 — live)

- **In-process pub/sub.** [apps/api/src/lib/message-bus.ts](../../apps/api/src/lib/message-bus.ts) is a thin EventEmitter keyed by `conversation_id`. Publish on insert, subscribe in the SSE route. Single-container design; Redis pub/sub is the obvious next step if we ever go multi-replica.
- **SSE route.** `GET /v1/conversations/{id}/events` (auth + participant check). Initial `event: ready` frame flushes headers + signals open. Then one `event: message` per publish with the full `Message` row as data. 25 s heartbeat keeps Caddy's default 60 s idle-proxy-timeout from killing quiet threads (bug-registry: `[Network] Long-Running Connection Dropped by Proxy / Load Balancer`).
- **Caddy pass-through.** [infra/Caddyfile](../../infra/Caddyfile) splits `api.{$DOMAIN}` into two `handle` blocks: `/v1/.../events` goes through `flush_interval -1` with no compression (gzip would buffer); everything else keeps `encode zstd gzip` for JSON throughput.
- **Same-origin proxy on the web.** [apps/web/app/api/messages/[id]/events/route.ts](../../apps/web/app/api/messages/[id]/events/route.ts) bridges browser `EventSource` (no header support) to the API's Bearer-authed endpoint. The proxy reads the session cookie server-side, opens the upstream with the access token, pipes the response body, and forwards client abort to the upstream fetch so the API-side cleanup fires.
- **Client.** [apps/web/app/messages/[id]/conversation-view.tsx](../../apps/web/app/messages/[id]/conversation-view.tsx) opens `EventSource('/api/messages/:id/events')` on mount. `ready` → catch-up fetch with the tail cursor (covers the page-render-to-SSE-open race); `message` → merge into state with the existing dedup. EventSource handles reconnect natively; on each reconnect the next `ready` event triggers another catch-up.
- **Fallback poll.** If `EventSource` never opens (network blocked, CDN buffering, proxy stripping), the same `fetchMessagesAfterAction` cursor fetch runs every 30 s. Both feeds share `mergeNew` so no dedupe handling needed at the consumer.

## Phase B-2 (live)

- **Active-view suppression.** `ACTIVE_VIEW_WINDOW_MS = 2 * 60 * 1000` in [apps/api/src/routes/v1/conversations.ts](../../apps/api/src/routes/v1/conversations.ts). The POST /messages handler reads both participants' `*_last_read_at` in the same pre-check that does the participant gate (one round-trip), then compares `now - recipient.last_read_at < window` to decide whether to email. NULL counts as "never seen" and always emails — the first message to a fresh thread always notifies.
- **Sender auto-mark-read.** The same POST handler bumps the SENDER's `*_last_read_at = now()` alongside `last_message_at`. Without this, a back-and-forth conversation would generate emails to both parties every turn — the sender is obviously looking at the thread when they hit Send.
- **Client side.** [apps/web/app/messages/[id]/conversation-view.tsx](../../apps/web/app/messages/[id]/conversation-view.tsx) fires `markConversationReadAction` on SSE `ready` (covers the "tab open, no new message yet" case) and on every incoming SSE message where `sender_id !== meId` (covers the "I'm reading right now while they're sending" case). The PATCH is idempotent + cheap, so we don't debounce; the server-side suppression check is what actually gates email send.
- **Email content.** [apps/api/src/lib/email.ts → sendNewMessageEmail](../../apps/api/src/lib/email.ts) sends `Subject: ${senderName}: ${preview80chars}`, body includes 240-char preview + a CTA button linking to `${WEB_ORIGIN}/messages/${conversation_id}`. Same fire-and-forget wrapping as the reserve-email path — any Resend failure logs + reports to Sentry, never blocks the POST response.
